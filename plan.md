# Plan

## Scope

全面检查 Minecraft 服务端下载、入库、检测、启动、配置读取这条链路，重点确认跨平台路径、列表刷新、首次启动行为是否完整。

## Findings

1. 启动参数曾存在重复拼接 JAR 路径的问题。
   - 现象：`java -jar` 后收到重复绝对路径。
   - 影响：macOS/Linux 上直接报 `Unable to access jarfile`。

2. 服务端列表刷新依赖页面首次挂载，状态同步不够主动。
   - 现象：下载完成后切回服务器管理页前，列表可能仍是旧数据。
   - 影响：用户感知为“需要重启软件才刷新”。

3. 首次启动前没有确保 `eula.txt` 处于可运行状态。
   - 现象：第一次启动服务端时会卡在 EULA。
   - 影响：需要手动同意，破坏一键启动体验。

4. 前端仍有少量 Windows 风格路径处理。
   - 现象：读取 `server.properties`、提取下载结果文件名时，之前存在硬编码反斜杠。
   - 影响：macOS/Linux 上容易出现读取失败或文件名识别错误。

5. Java 检测对 macOS 的覆盖不完整。
   - 现象：仅依赖 `java` 或 Linux 默认路径。
   - 影响：部分 macOS 安装方式下会误判“未检测到 Java”。

6. 启动前缺少 JAR 存在性校验。
   - 现象：JAR 文件缺失时，错误由 Java 进程抛出，提示不够直观。
   - 影响：排查成本较高。

## Completed

1. 修复 `server:start` 参数链路。
   - 主进程统一根据 `serverDir + jarName` 组装 JAR 路径。
   - 去掉渲染进程传递重复绝对路径的做法。

2. 修复服务端列表刷新机制。
   - 保留进入服务器管理页时的主动刷新。
   - 新增 `servers:changed` 事件，在新增、删除、更新服务器后主动通知前端刷新。

3. 增加首次启动前自动写入 `eula.txt`。
   - 若不存在或不是 `eula=true`，启动前自动修正。

4. 补齐跨平台路径处理。
   - `server.properties` 改为通用拼接逻辑。
   - 下载完成后改用同时支持 `/` 和 `\` 的 basename 提取方式。

5. 增强 Java 检测。
   - 加入 `JAVA_HOME`、`/usr/bin/java`、Homebrew OpenJDK、常见 macOS JDK 路径候选。

6. 增加启动前 JAR 校验。
   - 若目标文件不存在，直接输出清晰日志并标记为错误状态。

7. 清理配套类型声明。
   - 确保预加载接口扩展后，前端类型检查仍可通过。

## Verification

1. 运行 `npm run typecheck`，结果通过。

2. 建议手动回归以下场景：
   - macOS 下载 Vanilla 后直接启动，确认 `-jar` 路径不再重复。
   - 下载完成后切换到“服务器管理”，确认列表自动出现新服务器。
   - 首次启动空白服务端目录，确认自动生成 `eula.txt`。
   - 首次启动后重新进入“配置”页，确认 `server.properties` 可读取。
   - 手动删除目标 JAR 后点击启动，确认日志明确提示缺少 JAR 文件。

## Follow-up

1. 视需求决定是否将“下载目录”改为自动创建以服务器名命名的子目录。
   - 这样能避免多个服务端共用同一目录时互相覆盖。

2. 排查并统一项目源码文件编码。
   - 当前仓库存在中文乱码，不影响本次逻辑修复，但会影响维护体验。

3. 考虑在下载成功后自动跳转到服务器管理页并选中新建服务器。
   - 这会进一步降低“下载完成但不知道下一步做什么”的心智负担。
